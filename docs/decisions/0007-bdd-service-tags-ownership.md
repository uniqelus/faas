# 0007. BDD feature ownership через сервисные теги

- Статус: Superseded by 0008
- Дата: 2026-05-15

## Контекст

В `tests/integration` планируется BDD-контур на Godog/Cucumber. На старте были
варианты тегирования по типу проверок (`@smoke`, `@lifecycle`) и смешанной
структуры feature-файлов.

Проблема такого подхода:

- ownership сценариев размывается между сервисами;
- сложнее запускать целевые прогоны по конкретному сервису;
- при росте числа сценариев появляется смешение ответственности в одном feature.

Нужно зафиксировать базовое правило, которое не ломает масштабирование и
делает запуск/поддержку сценариев предсказуемыми.

## Решение

Для интеграционных BDD-тестов вводим сервисный срез как основной:

1. Feature-структура — по сервисам:
   - `tests/integration/features/api-gateway/`
   - `tests/integration/features/control-plane/`
2. Каждый feature-файл обязан иметь ровно один сервисный тег на уровне
   `Feature`: либо `@api-gateway`, либо `@control-plane`.
3. Смешивать сервисные ownership-теги в одном feature-файле запрещено.
4. Запуск интеграций фильтруется по `GODOG_TAGS`, в первую очередь сервисными
   тегами (например, `@api-gateway`).
5. Теги типа `@smoke`, `@regression`, `@negative` допускаются как вторичный
   срез на уровне `Scenario` или `Feature`, но не заменяют сервисный тег.

## Рассмотренные альтернативы

| Вариант | Почему не |
| --- | --- |
| Теги только по типу проверки (`@smoke`, `@lifecycle`) | Неочевидный ownership, сложнее сервисно-ориентированный запуск и ревью |
| Feature-структура по доменам + типам сценариев | Гибко, но повышает сложность навигации и не даёт жёсткого соответствия «сервис ↔ feature» |
| Без обязательного сервисного тега | Быстро на старте, но приводит к хаосу при расширении набора тестов |

## Последствия

**Положительные:**

- Явный ownership сценариев и проще ревью (видно, к какому сервису относится
  feature).
- Простой и стабильный запуск выборки: `GODOG_TAGS=@api-gateway` или
  `GODOG_TAGS=@control-plane`.
- Легче строить CI-матрицу по сервисам.

**Отрицательные:**

- Нужно дисциплинированно проверять наличие сервисного тега в каждом новом
  feature.
- Сценарии, затрагивающие оба сервиса, придётся размещать в одном ownership-
  сервисе и явно документировать границы ответственности.

## Триггеры пересмотра

- Появляется отдельный интеграционный слой для межсервисных end-to-end
  сценариев, где ownership по одному сервису перестаёт быть практичным.
- Количество cross-service сценариев стабильно превышает сервисные сценарии.
- CI начинает требовать другой основной срез запусков (например, только
  capability-based, а не service-based).

## Связанные решения

- ADR 0002 — Control API через gRPC + grpc-gateway.
- ADR 0006 — control-plane как plain API + Postgres (без CRD).
- ADR 0008 — полный дизайн интеграционных тестов на Godog, включающий это правило.
