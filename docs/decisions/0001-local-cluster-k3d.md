# 0001. Локальный кластер — k3d

- Статус: Accepted
- Дата: 2026-05-13

## Контекст

Платформе нужен Kubernetes-кластер для двух сценариев:

1. **Локальная разработка** одним инженером (быстрый цикл «снёс/поднял»,
   низкий overhead по RAM/CPU).
2. **CI** — прогонка интеграционных e2e-тестов в GitHub Actions без
   self-hosted runners.

Дополнительные требования:

- Загрузка локальных образов без внешнего registry (демо-функция и сервисы
  платформы билдятся локально).
- Ingress для проверки публичной поверхности gateway.
- Управление через Terraform, чтобы стенд был воспроизводим.

Правило репозитория `10-architecture.mdc` фиксирует `K3s` как дистрибутив по
умолчанию. В исходных требованиях MVP пользователь указал `minikube` —
противоречие нужно разрешить явно.

## Решение

Используем **`k3d`** — K3s, запущенный в Docker-контейнерах через утилиту
`k3d` (`https://k3d.io`). Управление через Terraform-модуль с `null_resource`
+ `local-exec`, без нативного провайдера (его нет в стабильном виде, и
обёртка над CLI достаточно идемпотентна через `triggers`).

Кластер именуется `faas`, проброс `--port 8080:80@loadbalancer` даёт
доступ к ingress через `localhost:8080`.

## Рассмотренные альтернативы

| Вариант | Плюсы | Минусы |
| --- | --- | --- |
| **minikube** | Самая популярная утилита, много документации, разнообразие драйверов | ~700 MB RAM idle, старт 30–60s, ingress через отдельный addon, в CI ощутимо медленнее |
| **kind** | Лёгкий, популярный в CI, простой image load | Это vanilla Kubernetes, не K3s — расходится с правилом репо; нет встроенного ingress |
| **K3s бинарь на хосте** | Самый быстрый, нет Docker-слоя | Нет изоляции кластеров, грязное удаление, конфликты с системой |
| **microk8s** | Хороший DX на Ubuntu | Snap-only, плохо ставится вне Ubuntu — не подходит как «по умолчанию» |
| **`k3d`** (выбран) | K3s в Docker → совпадает с правилом репо, ~250 MB RAM, старт <15s, `k3d image import`, встроенный Traefik, одинаково локально и в CI | Меньше документации, чем у minikube; нет нативного TF-провайдера |

Решение разрешает противоречие правил/требований: `k3d` — это **K3s**
(удовлетворяет правилу), но запущенный в Docker (удовлетворяет «локальному»
сценарию, в котором пользователь хотел `minikube`).

## Последствия

**Положительные:**

- Одинаковая среда локально и в CI → меньше «у меня работает».
- Быстрый цикл итераций (старт/снос <15s).
- Traefik встроен — не нужен отдельный ingress-controller subchart.
- Загрузка образов через `k3d image import` снимает требование к registry.

**Отрицательные:**

- Terraform-модуль реализован через `local-exec`, не через типизированный
  провайдер. Это значит: idempotency держится на `triggers` и проверках
  состояния, а не на самом TF.
- Меньше документации в сети, чем у minikube — на узких вопросах придётся
  читать исходники k3d.
- Требует Docker на хосте и в CI runner.

**Точечно:**

- В Helm-чарте платформы не используем `IngressClass: nginx` — используем
  встроенный Traefik k3d (или ставим nginx в чарт под флагом, если придёт
  необходимость).
- В Makefile цели `infra-up`/`infra-down` — обёртки над Terraform.
- Профиль `lite` (без observability-стека) важен особенно для CI, иначе
  кластер не помещается в стандартный GitHub runner по RAM.

## Триггеры пересмотра

- Если CI runners начнут падать по таймауту/RAM при поднятии `k3d` —
  рассмотреть `kind` (vanilla Kubernetes) и зафиксировать переход.
- Если появится нативный Terraform-провайдер для `k3d` со стабильным
  релизом — заменить `null_resource`-обёртку.
- Если целью станет максимально близкое к проду окружение для разработчиков
  (HA, embedded etcd, etc.) — переходить на полный K3s в VM/на хосте.
