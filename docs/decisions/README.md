# Architecture Decision Records

Каталог содержит ADR — короткие записи о значимых архитектурных решениях по
платформе. Каждый файл — одно решение.

## Зачем

- Чтобы новый человек (или ты сам через полгода) понимал, **почему** что-то
  сделано так, а не иначе, без раскопок в чатах и коммитах.
- Чтобы было видно границу: что обсуждалось, какие альтернативы отвергнуты,
  при каких условиях стоит пересмотреть решение.
- Чтобы изменения в архитектуре проходили через явный шаг «обновили или
  выпустили новый ADR», а не растворялись в коде.

## Формат

```
# NNNN. Заголовок

- Статус: Proposed | Accepted | Superseded by NNNN | Deprecated
- Дата: YYYY-MM-DD

## Контекст
Какую проблему решаем, какие ограничения, что важно.

## Решение
Что именно делаем. Один абзац или один список — без воды.

## Рассмотренные альтернативы
Чем ещё это можно было решить, и почему отвергли. Кратко.

## Последствия
Что меняется в коде/процессе. Плюсы и минусы. Что станет дороже.

## Триггеры пересмотра
Что должно произойти, чтобы решение перестало быть валидным.

## Связанные решения
Ссылки на другие ADR.
```

## Правила

1. **Номер растёт монотонно.** Не переиспользуем, не вставляем «в середину».
2. **Не редактируем принятый ADR**, кроме как для смены статуса
   (`Superseded by NNNN`). Изменение решения = новый ADR, который явно
   ссылается на предыдущий.
3. **ADR пишется до или вместе с кодом**, не задним числом по факту.
4. **Файл — самодостаточен.** Не нужно лезть в чат или в код, чтобы понять,
   о чём он.
5. **PR с архитектурными изменениями** проходит ревью отдельно от PR с кодом
   реализации этого решения. Сначала ADR — потом код.
