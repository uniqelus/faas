# 0008. Дизайн интеграционных тестов на Godog

- Статус: Accepted
- Дата: 2026-05-15

## Контекст

Проекту нужен интеграционный тестовый контур в `tests/` на базе Cucumber/Gherkin
с исполнителем Godog.

Цели первой итерации:

- быстрый локальный запуск без k3d/helm (сервисы как локальные процессы);
- понятный ownership тестов по сервисам;
- переиспользуемые шаги для HTTP/gRPC/SQL и дальнейшего роста сценариев;
- управляемый запуск через существующий `make test`, без отдельного обязательного
  таргета.

## Решение

### 1. Структура тестов

Базовый каталог:

- `tests/integration/`

Внутри:

- `features/` — feature-файлы, разложенные по сервисам:
  - `features/api-gateway/`
  - `features/control-plane/`
- `steps/` — переиспользуемые шаги по области применения:
  - `http_steps.go`
  - `grpc_steps.go`
  - `sql_steps.go`
  - `process_steps.go`
  - `registry.go` (единая регистрация шагов в `ScenarioContext`)
- `harness/` — инфраструктурные хелперы:
  - `runtime.go` (запуск/остановка subprocess для бинарей)
  - `http_client.go`
  - `grpc_client.go`
  - `sql_client.go`
  - `assertions.go`
- `logs/` — общие логи suite (создаётся во время запуска тестов).

Дополнительно в сервисных feature-папках допускаются:

- `testdata/` для локальных payload/fixtures и test runtime config;
- `logs/` для локальных диагностических данных.

### 2. Правило ownership и тегов

- Каждый feature-файл обязан иметь ровно один сервисный тег на уровне `Feature`:
  `@api-gateway` или `@control-plane`.
- Смешение сервисных ownership-тегов в одном feature-файле запрещено.
- Сервисные теги — основной срез запуска.
- Дополнительные теги (`@smoke`, `@regression`, `@negative`) — вторичный срез и
  не заменяют сервисный ownership.

### 3. Модель запуска

- Интеграционный запуск выполняется через флаг:
  - `INTEGRATION=1 make test`
- Без `INTEGRATION=1` поведение `make test` остается текущим (lint + unit/component).
- В интеграционном режиме запуск выполняется как `go test -v -count=1` для
  явного вывода Godog-сценариев и отключения test cache.
- Для выбора набора сценариев используется:
  - `GODOG_TAGS` (например, `@api-gateway`, `@control-plane`).

### 4. Тестовый жизненный цикл

- `BeforeSuite`: инициализация suite-контекста и hooks.
- `BeforeScenario`: подготовка сценарного контекста, генерация test-id.
- Старт сервисов выполняется явно в шагах `Given ... is running`, с readiness
  проверкой `/healthz`.
- `AfterSuite`: graceful shutdown subprocess с timeout и fallback-kill.
- Логи сервисов пишутся в `tests/integration/logs/`, каталог создаётся только
  во время прогона.
- Файлы логов сервисов перезаписываются на каждом запуске (без накопления от
  предыдущих прогонов).

### 5. Сценарии первой итерации

- `api-gateway`:
  - health/metrics smoke (`/healthz`, `/metrics`);
  - baseline lifecycle-роуты через API gateway (create/get/list/delete) на
    текущем этапе проверяются как доступные endpoint-ы (ожидаемый `404` до
    подключения handler-ов бизнес-логики).
- `control-plane`:
  - доступность gRPC-порта;
  - admin endpoints (`/healthz`, `/metrics`).

## Рассмотренные альтернативы

| Вариант | Почему не |
| --- | --- |
| Отдельный `make test-integration` | Разводит точки входа, при этом команда предпочитает параметризованный `make test` |
| Теги только по типу сценариев (`@smoke`, `@lifecycle`) | Размытый ownership и менее удобный сервисный запуск |
| Плоская структура feature без сервисных папок | Плохая масштабируемость и сложнее навигация |

## Последствия

**Положительные:**

- Единая и предсказуемая структура `tests/integration`.
- Простое масштабирование по сервисам и по типам сценариев.
- Переиспользуемые step-модули снижают дублирование.
- Четкий service ownership для ревью и CI-матриц.

**Отрицательные:**

- Выше дисциплина по тегам и структуре feature-файлов.
- Нужна поддержка harness-кода и артефактной диагностики.
- Cross-service сценарии требуют явного выбора owner-сервиса.

## Триггеры пересмотра

- Появляется полноценный k3d/helm e2e-слой как основной путь интеграций.
- Доля межсервисных сценариев становится существенно больше сервисных.
- Команда решает разделить быстрые integration и тяжелые e2e по разным
  отдельным workflow/репозиториям.
