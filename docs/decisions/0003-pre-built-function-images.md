# 0003. Функции — pre-built OCI-образы

- Статус: Accepted
- Дата: 2026-05-13

## Контекст

Функция в нашем FaaS — это HTTP-обработчик, который что-то делает в ответ на
запрос. Чтобы её задеплоить в Kubernetes, нужен контейнерный образ. Есть
несколько способов получить этот образ:

1. Пользователь приносит готовый образ (`image: my-registry/fn:tag`).
2. Платформа собирает образ из исходников (Buildpacks, kaniko, BuildKit).
3. Платформа поддерживает несколько шаблонов рантаймов (Go, Python, Node) и
   собирает на основе шаблона.

Варианты 2 и 3 — это отдельные подсистемы со своим масштабом: registry,
job-ы сборки, очистка артефактов, кеши, build-cache invalidation,
многослойный security review (что в `RUN apt install …`).

Стоит вопрос: что входит в MVP?

## Решение

Платформа принимает **pre-built** OCI-образ. На уровне API control-plane:

- Валидирует ссылку на образ простым regex.
- Прокидывает `imagePullSecrets` через конфиг платформы (опционально, для
  приватных registry).
- Не выполняет ни сборки, ни валидации содержимого образа.

Эталонная функция (`examples/hello-go`) собирается обычным
`docker build` + `k3d image import` — это и есть рекомендованный workflow
для пользователя в MVP.

Контракт функции:

- Контейнер слушает HTTP на порту, заданном через переменную окружения `PORT`
  (по умолчанию `8080`).
- Отвечает на `GET /healthz` (readiness/liveness probe).
- Отвечает на любой путь, который вызовет пользователь через
  `POST /fn/{name}/...`. Маршрутизацию внутри функции организует разработчик
  функции.

## Рассмотренные альтернативы

| Вариант | Плюсы | Минусы |
| --- | --- | --- |
| **Buildpacks (paketo)** | Хорошее DX (`pack build`), не нужен Dockerfile | +~1 неделя в скоупе MVP, нужно managed-buildkit в кластере, плюс кеши слоёв, плюс registry |
| **kaniko/BuildKit job в кластере** | Очень гибко, native K8s | Требует регистри, RBAC, очистку job-ов, security review (build runs as root в namespace) |
| **Шаблоны рантаймов с готовыми образами** | Минимум кода от пользователя | Каждый рантайм — свой Dockerfile, своё CVE-обновление; для MVP избыточно |
| **Внешний CI пользователя билдит и пушит** (выбран) | Простота, чёткая граница ответственности | Пользователь обязан уметь в Dockerfile |

## Последствия

**Положительные:**

- Скоуп control-plane сокращается на ~1.5–2 недели работы.
- Нет registry-зависимости. Для локального demo достаточно
  `k3d image import`. Для прода — любой OCI-registry, который умеет
  Kubernetes.
- Чёткая граница: платформа отвечает за lifecycle и observability функции,
  не за её упаковку.

**Отрицательные:**

- Порог входа выше: пользователь должен знать, как написать Dockerfile.
- Невозможно демонстрировать «git push → live function» без внешнего CI.
- Нужно где-то документировать контракт функции (`PORT`, `/healthz`, HTTP).

**Точечно:**

- `examples/hello-go/Dockerfile` — публичный референс контракта.
- В документации платформы — раздел «Function Contract» с минимальными
  требованиями к образу.
- В API control-plane `Function.image` валидируется regex-ом, но содержимое
  не проверяется (`ImagePullBackOff` ловим уже в status функции через
  реконсайлер).

## Триггеры пересмотра

- Если в Phase 2 пользователи попросят «нативный язык, не хочу думать про
  Dockerfile» — добавляем Buildpacks как опциональный путь поверх той же API
  (поле `source` вместо `image`).
- Если набьются десятки функций и появится боль с тем, что у каждой свой
  Dockerfile и свой process-cycle — рассматриваем шаблоны рантаймов.

## Связанные решения

- ADR 0006 — control-plane никак не знает про процесс сборки; это решение
  упрощает реконсайлер.
